暑期科研项目总结

暑期科研项目总结

本次项目的主要内容

本次暑期科研项目，主要实现了基于A3C算法的智能体完成星际争霸枪兵护互拼的研究。

本次项目的难点分析

在利用深度强化学习玩游戏的研究中，星际争霸2中agent还没有取得好的成绩。原因如下：

  • 连续动作控制，每个动作之间有相关性，要从相关的一系列指令中学习

  • agent所处的环境更加复杂，有了战争迷雾的存在，对局势的估计更加的复杂

  • Reward难以估计

先前准备与学习

对深度学习的基础知识和tensorflow等代码实现有了基本的了解。

通过morvan的强化学习教程，深入学习了A3C算法等一系列Actor-Critic算法。

项目中个人工作和遇到的问题

工作概述

  • 学习DDPG算法，并做总结梳理

    在项目开始的第一周，我负责学习DDPG算法和梳理整理DDPG算法的特性。
    并考虑使用DDPG算法实现相同的功能与A3C版作对比。

  • 参与Reward的设计构思

    参与构思Reward的设计。对于上层网络和下层网络而言，暴雪分数并不是一个合理的Reward。我们考虑上层网络是战略决策网络，而下层网络是战术决策网络。（策略分层），并设计不同的Reward。

    对于一个agent来说，在星际争霸的环境中，实际上是资源抢夺和造兵攻击的零和博弈。为了获取胜利其实要不断提高以下几个比例。在网络的设计中，可以考虑从这三个比例中着手设计Reward。

    1. 资源转化率->当前已有资源转化为兵力的比例。
    2. 输出率->对地方造成输出的和损失兵力的比例。

    3. 获益率->通过给对方造成伤害，和获得了利益衡量的比例。

      考虑到agent获取胜利的难度较大，我们考虑通过18分钟的时间作为限制，让agent在与电脑的对战中坚持更长的时间。同时在攻击地方和获得游戏胜利中给予更高的Reward。

  • 代码的合并与调试

    与董浩楠同学一起整理项目的代码，并运行出最初的一些成果。

问题总结

  • 原版代码的loss问题

    原版代码的损失函数定义为`loss = value_loss + target_loss`,然后通过`dl/dw`更新参数，实际上是有问题的。发现问题后，我和董浩楠同学决定，对网络进行修改，分离Actor和Critic网络，这样在更新参数的时候避免了问题，同时提高了训练速度。

  • tensorflow中scope.reuse的问题

    `tf.get_variable()`会结合`reuse`进行检查返回tensor
`tf.placeholder()`和`tf.variable()`不会检查，而是遇到冲突，用命名后加`_1`区分

  • 在A3C算法中利用local net的梯度，更新global net，而不是将local net的参数push给global。原版代码中这一部分有很大的问题。

本次项目的收获

  • 通过本次项目经历，我学习到了很多深度学习和强化学习领域的前沿知识。对从未涉足过的一些深度学习算法有了更加深刻的理解和认识。从整个科研项目过程中，体验了科研工作的流程。

  • 在项目开发过程中，我同时学习了deeplearning.ai的深度学习教程，不断夯实自己的基础知识。对深度学习有了更加深入的理解和认识。

问题和启发

未解决的问题

  • 使用DDPG算法实现类似功能的Agent，并与A3C版进行性能比较，希望可以在本次研究结束后，抽出一部分时间，完成DDPG算法的实现。

避免再犯的问题

  • 本次项目维护代码的过程中，浪费了大量的经历。在一个小组协同开发的项目中，
    做好接口的定义和适当的文档说明非常重要。
  • 对于代码的维护，建议使用SVN/Git等源代码管理工具，以提高效率和方便对比。

DDPG算法总结

Posted on 2018-08-21 | Edited on 2018-09-08

DDPG算法总结

背景描述

RL要解决的问题是：让agent学习在一个环境中的如何行为动作(act)， 从而获得最大的奖励值总和(total reward)。这个奖励值一般与agent定义的任务目标关联。agent需要的主要学习内容：第一是行为策略(action policy)， 第二是规划(planning)。其中，行为策略的学习目标是最优策略， 也就是使用这样的策略，可以让agent在特定环境中的行为获得最大的奖励值，从而实现其任务目标。

行为(action)可以简单分为：

  • 连续的：如赛车游戏中的方向盘角度、油门、刹车控制信号，机器人的关节伺服电机控制信号等

  • 离散的：如围棋、贪吃蛇游戏。 Alpha Go就是一个典型的离散行为agent。

DDPG是针对连续行为的策略学习方法。

DDPG

Deepmind在2016年提出DDPG，全称是：Deep Deterministic Policy Gradient。是将深度学习神经网络融合进DPG的策略学习方法。
相对于DPG的核心改进是：采用 卷积神经网络作为策略函数μ和Q函数的模拟，即策略网络和Q网络；然后使用深度学习方法来训练上述神经网络。

Q函数的实现和训练方法，采用了Deepmind 2015年发表的DQN方法。

online网络与target网络

以往的实践证明，如果只使用单个”Q神经网络”的算法，学习过程很不稳定，因为Q网络的参数在频繁gradient update的同时，又用于计算Q网络和策略网络的gradient。

基于此，DDPG分别为策略网络、Q网络各创建两个神经网络拷贝,一个叫做online，一个叫做target:

在训练完一个mini-batch的数据之后，通过随机梯度下降更新online网络的参数，然后再通过soft update算法更新 target 网络的参数。soft update是一种running average的算法：

也就是每次target网络参数的更新，是target网络参数和online网络参数的线性组合。这样做的优点和缺点都很明显。

优点：target网络参数变化小，用于在训练过程中计算online网络的gradient，比较稳定，训练易于收敛。

代价：参数变化小，学习过程变慢。

算法流程

首先代码的第一部分是初始化。

  • 初始化actor\critic的 online 神经网络参数: θQ 和 θμ
  • 将online网络的参数拷贝给对应的target网络参数：θQ′←θQ,θμ′←θμ ;

  • 初始化replay memory buffer R;

  • 然后对每个回合进行训练。（第一个for）

    • 每个回合要先初始化随机噪声N(图中的希腊字母)
    • 获得初始的状态s1
    • 然后对每一个time step进行训练（第二个for）
      • 先从根据online策略网络和噪声获得输出的动作at，at=μ(st|θμ)+Nt
      • 执行该动作，从环境中获取reward和下一个状态的St+1
      • 将这个动作转换存储到经验池（memery buffer）R中
      • 从R中随机采样N个transition数据即 (st,at,rt，st+1)，作为训练online-actor,online-critic网络的训练数据。
      • 根据target网络获得估计的Q值，定义loss为MSE: mean squared error：L=(1/N)∑i(yi−Q(si,ai|θQ))^2(见上图吧)用这个Q值更新online-critic网络(ps:这里可以看成监督学习的方法，yi是标签）
      • 基于标准的bp使用梯度下降的策略更新online-actor网络
      • 使用soft update的方法更新target-critic,target-actor网络

有两点值得注意：

  • yi 的计算，使用的是 target 策略网络μ′ 和 target Q 网络Q′,这样做是为了Q网络参数的学习过程更加稳定，易于收敛。

  • 这个标签本身依赖于我们正在学习的target网络，这是区别于监督式学习的地方。update online Q： 采用Adam optimizer更新θQ;

DDPG对于DPG的关键改进

  1. 使用卷积神经网络来模拟策略函数和Q函数，并用深度学习的方法来训练，证明了在RL方法中，非线性模拟函数的准确性和高性能、可收敛；而DPG中，可以看成使用线性回归的机器学习方法：使用带参数的线性函数来模拟策略函数和Q函数，然后使用线性回归的方法进行训练。

  2. experience replay memory的使用：actor同环境交互时，产生的transition数据序列是在时间上高度关联(correlated)的，如果这些数据序列直接用于训练，会导致神经网络的overfit，不易收敛。DDPG的actor将transition数据先存入experience replay buffer, 然后在训练时，从experience replay buffer中随机采样mini-batch数据，这样采样得到的数据可以认为是无关联的。

  3. target网络和online网络的使用， 使的学习过程更加稳定，收敛更有保障。

pysc2项目详解

Actions

Observations

observation_spec返回的是一个tuple。tuple每一个元素是一个tuple构成的dict，对应于每一个agent。

每一个agent的observation包括Spacial

Feature layers

在 pysc2.lib.features 中定义

  1. feature_screen -> 13个

    • height_map
    • visibility
    • creep
    • power
    • player_id
    • player_relative
    • unit_type
    • selected
    • hit_points
    • energy
    • shields
    • unit_density
    • unit_desity_aa

  2. feature_minimap -> 7个

    • height_map
    • visibility
    • creep
    • camera
    • player_id
    • player_relative
    • selected

Structure elements

  1. General player information
  • player_id
  • minerals
  • vespene
  • food used (otherwise known as supply)
  • food cap
  • food used by army
  • food used by workers
  • idle worker count
  • army count
  • warp gate count (for protoss)
  • larva count (for zerg)
  1. Control groups

A (10, 2) tensor showing the (unit leader type and count) for each of the 10
control groups. The indices in this tensor are referenced by the control-group
action.

Control groups are a way to
remember a selection set so that you can recall them easily later.

  1. Single Select

A (7) tensor showing information about a selected unit.

  • unit type
  • player_relative
  • health
  • shields
  • energy
  • transport slot taken if it’s in a transport
  • build progress as a percentage if it’s still being built
  1. Multi Select

A (n, 7) tensor with the same as single select but for all
n selected units. The indices in this tensor are referenced by the
select_unit action.

  1. Cargo

A (n, 7) tensor similar to single select, but for all the
units in a transport. The indices in this tensor are referenced by the unload
action.

  1. Build Queue

A (n, 7) tensor similar to single select, but for all the
units being built by a production building. The indices in this tensor are
referenced by the build_queue action.

  1. Available Actions

A (n) tensor listing all the action ids that are available at the time of this
observation.

  1. Last Actions

A (n) tensor listing all the action ids that were made successfully since the
last observation. An action that was attempted but failed is not included here.

  1. Action Result

A (n) tensor (usually size 1) giving the result of the action. The values are
listed in
error.proto

  1. Alerts

A (n) tensor (usually empty, occasionally size 1, max 2) for when you’re being attacked in a major way.
